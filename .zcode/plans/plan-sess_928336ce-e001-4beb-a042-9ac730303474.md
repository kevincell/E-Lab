## Goal
Add Python, Java, and C++ compilation/execution to the e-Lab sandbox (currently C-only), with reliable, clearly-surfaced error messages for compile errors, runtime errors, and TLE — on both the **Run** panel and the **Submit** results page.

**Java class handling (your decision): auto-detect.** The runner scans the source for `public class X` (fallback: first `class X`) and saves the file as `<X>.java` so it always compiles under the correct name. Students never see a "filename doesn't match class name" error regardless of what they name their class.

## How everything routes today
All execution goes through one function: `core/sandbox.py::run_c_code()`, called by:
- `services.py::evaluate_submission()` → the **Submit** path (Celery task)
- `views.py::run_code()` → the **Run** button

It spins up a throwaway `docker run` against the `elab-sandbox` image (`sandbox/Dockerfile`, currently just `FROM gcc:latest`). Compile + run happen in a single `sh -c`, and compile errors are guessed by searching for `"error:"` in output — fragile and it loses the real compiler message.

**New design:** one `run_code(language, ...)` dispatcher backed by a `LANGUAGES` registry; one multi-language sandbox image; and a marker-based two-phase inner script so compile-vs-runtime failure is detected definitively (no substring guessing) and the real compiler message is always preserved.

## Files to change

### 1. `sandbox/Dockerfile` — add Python + Java to the single image
Keep one image (so `check_sandbox()` and docker-compose are untouched):
```dockerfile
FROM gcc:latest
RUN apt-get update && apt-get install -y --no-install-recommends \
        python3 openjdk-17-jdk-headless && rm -rf /var/lib/apt/lists/*
ENV JAVA_HOME=/usr/lib/jvm/java-17-openjdk-amd64
ENV PATH="${JAVA_HOME}/bin:${PATH}"
WORKDIR /box
```
(gcc image already provides `gcc` + `g++`.)

### 2. `core/sandbox.py` — rewrite the engine (biggest change)
- Add a `LANGUAGES` registry. Each entry has: source filename, compile command (or `None` for interpreted), run command, and the Monaco editor language id.
  - `c` → `main.c`, `gcc -Wall -Wextra -Werror=return-type -Werror=main -std=c11 main.c -o program`, run `./program`
  - `cpp` → `main.cpp`, `g++ -Wall -Wextra -Werror=return-type -Werror=main -std=c++17 main.cpp -o program`, run `./program`
  - `python` → `main.py`, no compile, run `python3 main.py`
  - `java` → auto-detect class name → `<Name>.java`, compile `javac <Name>.java`, run `java <Name>`. Detection: regex `public\s+class\s+(\w+)` first, fallback `class\s+(\w+)`.
- Add `language_for_id(language_id)` mapping Judge0-style ids: 50→c, 54→cpp, 62→java, 71→python (50 stays default).
- New `run_code(language, source_code, stdin, expected_output, time_limit, memory_limit_kb)` that:
  1. Writes source + `input.txt` into a tmp box dir.
  2. For Java: regex-scans for the class name to pick the filename.
  3. Runs one `docker run --rm --network none ...` with an inner script that **separates compile and run**:
     ```sh
     cd /box
     <compile> >compile.out 2>&1 || { echo __COMPILE_ERROR__; cat compile.out; exit 3; }
     timeout <TL>s <run> <input.txt
     ```
  4. Classifies definitively:
     - marker/exit 3 → **Compilation Error** (status 6) with the real compiler output.
     - exit 124 → **Time Limit Exceeded** (status 5).
     - non-zero exit → **Runtime Error** (status 11), include stderr.
     - else compare to expected → Accepted / Wrong Answer.
  5. Returns dict: `status_id, status, stdout, stderr, compile_output, time, memory` (unchanged shape, but `compile_output`/`stderr` now reliably populated).
- Keep `run_c_code(...)` as a thin wrapper → `run_code("c", ...)` so nothing breaks if a call site is missed.

### 3. `core/services.py::evaluate_submission` — use dispatcher, short-circuit compile errors
- Derive language from `submission.language_id` (fallback `question.language_id`) via `language_for_id()`.
- Replace `run_c_code(...)` with `run_code(language, ...)`.
- **Short-circuit:** if a test returns `COMPILE_ERROR`, store the message in `submission.error_output`, set status, and stop (no point re-running an identical failing compile N times).
- Add `compile_output`/`stderr` into each `test_results` entry so the results page can show per-test errors.

### 4. `core/views.py::run_code` (Run button) — route by language + return rich errors
- Derive language from `question.language_id`; call `run_code(language, ...)`.
- Return each test result with `status` **and** an `error` field (compile/runtime message) so the Run panel can show "Compilation Error: …" instead of an empty "Your Output".

### 5. `templates/student/question_detail.html` — language-aware editor + error rendering
- View passes `question_language` (c/cpp/python/java) + dynamic filename.
- Monaco `language:` switches to it; `main.c` filename label becomes dynamic.
- Extend `renderTestResults()`: when a test has an `error`, render a red error block with the real message (compile output / stderr) instead of silently empty output.

### 6. `core/forms.py::QuestionForm` — friendly language dropdown

## Strictness summary (LeetCode-like)
- C/C++: `-Wall -Wextra` for visibility; `-Werror=return-type` and `-Werror=main` make **missing `return`** and **malformed `main`** hard errors (your explicit example). Benign warnings (unused vars) are shown, not fatal.
- Java: missing return is already a `javac` error natively; class-name mismatches eliminated by auto-detect.
- Python: SyntaxError / tracebacks returned as readable messages.
